# Workflow Rules

## Journal System

All project knowledge is stored as a graph of journal entries in `journal/`. The graph uses Obsidian-style `[[wikilinks]]` for cross-references. Each entry is a node; wikilinks are edges.

### Entry Structure

Each journal entry is a directory under `journal/` containing:

- `decisions.md` — **required for all entries**. Contains YAML frontmatter (title, type, status, created, tags, depends-on, informs) and the current truth about decisions made. This is the canonical source of what was decided and why.
- `plan.md` — implementation entries only. Phased implementation plan.
- `progress.md` — implementation entries only. Updated after every major milestone.
- `research/` — subfolder for research findings. Each finding is a separate markdown file.

### Entry Types (frontmatter only, NOT folder structure)

All entries live flat under `journal/` regardless of type. Type is a metadata field in `decisions.md` frontmatter, NOT a directory grouping. The index groups entries by type for readability, but the filesystem is flat.

- **vision** — big picture: product goals, positioning, strategy
- **implementation** — a specific feature or system to build (has plan.md + progress.md)
- **research** — completed investigation or codebase exploration
- **reference** — stable background knowledge (rarely changes)

Example: `journal/vision/` is an entry named "vision", NOT a folder containing vision-type entries. `journal/provider-architecture/` is an implementation-type entry that lives at the same level.

### Obsidian Wikilinks

All cross-references between journal entries MUST use Obsidian `[[wikilinks]]` syntax:

- `[[entry-name]]` — link to another journal entry's decisions.md
- `[[entry-name/plan]]` — link to a specific child file
- `[[entry-name/research/finding-name]]` — link to a specific research file

Use wikilinks in:

- Frontmatter `depends-on` and `informs` arrays
- Inline body text when referencing related decisions or research
- This enables Obsidian's graph view to visualize the knowledge topology

### The Index

`journal/_index.md` is the compiled graph. It is auto-generated by `/journal-validate` and contains:

- Every entry grouped by type (Vision, Implementation, Research, Reference)
- Implementation entries show their child nodes (plan, progress, research files) indented
- One-line description, status, and tags for each entry
- The agent reads this at session start to understand what knowledge exists

### During Work

- When starting any task: invoke `/journal-start <name>` to create or extend a journal entry
- Update `progress.md` after every major milestone (for implementation entries)
- Update `decisions.md` when a decision is made or changed
- Add `[[wikilinks]]` when referencing other entries
- When a decision supersedes an old one, mark the old entry's status as `stale`

### On Context Compaction

- If context was compacted, tell the user to run `/remember <feature>` to reload relevant journal context
- Always read `journal/_index.md` first to understand the graph before loading specific entries

### Saving New Knowledge

When significant decisions or insights emerge from a conversation:

1. Identify which journal entry they belong to (or create a new one)
2. Update `decisions.md` with the new decisions
3. Add wikilinks to related entries
4. Update `journal/_index.md` to reflect any new entries or status changes

## Commit Conventions

- Format: `feat/fix/chore/docs: <short message>`
- Never mention AI/Claude in commits
- Never push without explicit user instruction
